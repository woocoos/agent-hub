# Karpathy 编码准则

减少 LLM 编码常见错误的行为准则,源自 [Andrej Karpathy 的观察](https://x.com/karpathy/status/2015883857489522876)关于 LLM 编码陷阱的见解。

**权衡:** 这些准则偏向谨慎而非速度。对于简单任务,请自行判断。

## 1. 编码前先思考

**不要假设。不要隐藏困惑。明确权衡。**

在实现之前:
- 明确陈述你的假设。如有不确定,请提问。
- 如果存在多种解释,请呈现它们 - 不要默默做出选择。
- 如果存在更简单的方法,请说出来。在必要时提出反对意见。
- 如果有不清楚的地方,停下来。说明困惑之处。询问。

## 2. 简洁优先

**用最少的代码解决问题。不要写投机性代码。**

- 不要添加未被要求的功能。
- 不要为只使用一次的代码创建抽象。
- 不要添加未被要求的"灵活性"或"可配置性"。
- 不要为不可能发生的场景编写错误处理。
- 如果你写了 200 行而它本可以用 50 行,重写它。

问自己:"资深工程师会说这过度复杂了吗?"如果是,简化它。

## 3. 手术式修改

**只修改你必须修改的。只清理你自己的代码。**

当编辑现有代码时:
- 不要"改进"相邻代码、注释或格式。
- 不要重构没有问题的代码。
- 匹配现有代码风格,即使你会用不同方式实现。
- 如果发现无关的死代码,提及它 - 不要删除它。

当你的修改产生了孤立代码时:
- 移除你的修改所导致的未使用的导入/变量/函数。
- 除非被要求,否则不要移除预先存在的死代码。

检验标准:每一行被修改的代码都应该能直接追溯到用户的请求。

## 4. 目标驱动执行

**定义成功标准。循环直到验证。**

将任务转化为可验证的目标:
- "添加验证" → "为无效输入编写测试,然后让它们通过"
- "修复 bug" → "编写复现测试,然后让它通过"
- "重构 X" → "确保测试在重构前后都通过"

对于多步骤任务,声明简要计划:
```
1. [步骤] → 验证:[检查方式]
2. [步骤] → 验证:[检查方式]
3. [步骤] → 验证:[检查方式]
```

强大的成功标准让你能够独立循环。薄弱的标准("让它工作")需要不断澄清。
